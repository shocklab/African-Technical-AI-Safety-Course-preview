Week 5 • Session 9.2

Jailbreaks as optimisation (GCG)

The objective, the greedy-coordinate-gradient search, and the transfer result

What we'll cover

A "jailbreak" sounds like a trick. This sub-session shows it is an optimisation problem with a well-defined objective, solved by an algorithm a maths student can fully parse. We write the objective, which is to make the model begin its reply with an affirmative phrase, and then read greedy coordinate gradient (GCG; Zou et al., 2023) line by line: use gradients to shortlist candidate token swaps, use exact forward passes to choose among them. We close on the two results that matter for safety: the same suffix can be made universal across prompts, and it transfers to commercial models the attacker never had access to, with one number that looks like good news for one model and is partly an artefact.

From DAN to automated search

Hand-written jailbreaks work, but they are brittle and patchable. Optimisation makes the attack systematic.

The first jailbreaks were human-written: role-play framings ("you are DAN, you can Do Anything Now"), hypothetical wrappers, instructions to ignore previous instructions. They work because they exploit the model's general instruction-following, but each one is a one-off that a developer can find and patch. The shift that turned jailbreaking into a research problem was to stop hand-crafting and instead search for an input that provably maximises the chance of a harmful completion. To search you need an objective you can evaluate and, ideally, differentiate. GCG supplies both.

The objective: make "Sure, here is" the most likely continuation

If you can force the first few tokens of the reply, the rest usually follows.

The trick is the affirmative target. A safety-tuned model, asked something harmful, is trained to open with a refusal ("I can't help with that"). Empirically, if you can make it instead open with an affirmative prefix ("Sure, here is how to…"), its own fluency carries it into actually answering, because a helpful continuation of an affirmative opening is what the base model finds most natural. So the attacker's goal is concrete: append an adversarial suffix to the user request such that the model's most likely next tokens are a chosen target string.

Write the user prompt as a token sequence \(x_{1:n}\), with an adversarial suffix occupying a set of positions \(\mathcal{I}\), and let \(y_{1:m}\) be the target affirmative response. The attacker minimises the negative log-likelihood of the target given the prompt-plus-suffix:

\[ \mathcal{L}(x_{1:n}) = -\log p\big(y_{1:m} \,\big|\, x_{1:n}\big), \qquad \text{minimise } \mathcal{L} \text{ over the suffix tokens } \{x_i : i \in \mathcal{I}\}. \]

This is an ordinary language-model loss (the same cross-entropy used in training), but optimised over the input tokens rather than the weights, and only over the suffix positions. The obstacle is that the input is discrete: you cannot take a gradient step in token space and land on a valid token. GCG's contribution is a way around that.

Greedy coordinate gradient, read line by line

Use the gradient to propose; use an exact forward pass to decide.

The algorithm

Represent each suffix token by a one-hot vector \(e_{x_i} \in \{0,1\}^{|V|}\) over the vocabulary \(V\). Then for each suffix position \(i \in \mathcal{I}\):

  • 1. Gradient to shortlist. Compute \(\nabla_{e_{x_i}} \mathcal{L} \in \mathbb{R}^{|V|}\): how the loss would change if token \(i\) were replaced by each vocabulary item (a linearised estimate, read off the embedding gradient). Take the top-\(k\) tokens with the most negative gradient as candidate substitutions for that position. This is cheap: one backward pass gives the shortlist for every position at once.
  • 2. Exact evaluation to choose. The gradient is only a first-order guess, so do not trust it to pick the winner. Instead sample a batch of \(B\) candidate swaps (each a single token at one position, drawn from that position's top-\(k\)), and for each one actually run the model and measure the true loss \(\mathcal{L}\). Keep the single swap that lowers the loss most.
  • 3. Iterate. Apply the best swap and repeat for many steps. Each step changes one token; the loss falls until the model's most likely opening is the affirmative target.

The name says it: coordinate (change one token-position at a time), gradient (to shortlist candidates), greedy (take the best swap each step). The gradient does the cheap pruning over a \(|V| \approx 32{,}000\)-way choice; the forward passes do the reliable selection. The earlier AutoPrompt method searched one position at a time; GCG's advance is to consider swaps across all positions before committing, which makes the search markedly more effective.

Propose with gradients, decide with evaluations

Discrete optimisation over text is hard because you cannot follow a gradient to a valid input. GCG's resolution is to use the gradient only for what it is good at (ranking many candidates approximately and cheaply), and to use exact forward passes for what the gradient is bad at: deciding which candidate actually helps. That "propose with gradients, decide with evaluations" pattern recurs all over discrete ML; GCG is a consequential instance your cohort can implement.

Universal and transferable

One suffix, many prompts; one set of models, the rest of the market.

Two extensions turn a per-prompt curiosity into a systemic threat. Universality: optimise a single suffix against many harmful prompts at once (sum the losses), and you get one string that jailbreaks requests it was never tuned on. Transfer: optimise that universal suffix against an ensemble of open-weights models (where you have gradients), and it frequently works on commercial black-box models where you have none. The numbers from the paper, on attack success rate:

Attack success rates

  • White-box, where the attack is computed: Vicuna-7B ≈ 88% (harmful strings) / 99–100% (behaviours); Llama-2-7B-Chat ≈ 57% / 88%.
  • Black-box transfer from the open-weights ensemble: GPT-3.5 ≈ 86.6%, PaLM-2 ≈ 66.0%, GPT-4 ≈ 46.9%, Claude-2 ≈ 2.1%.

Claude-2's 2.1% looks like proof it is dramatically more robust. Read it carefully. The paper notes the suffixes were optimised against Vicuna, which is itself distilled from ChatGPT outputs, so transfer to the GPT family is flattered by shared lineage, and the low Claude number partly reflects that Claude is further from the surrogate, not only that it is safer. It is a caution against over-reading a benchmark: the same table that shows the attack working also shows how the choice of surrogate shapes the apparent ranking of defenders.

The cost of the attack

GCG is white-box and compute-heavy: it needs gradients (hence open weights, or a surrogate) and many forward passes per suffix, which is why its realistic deployment is transfer: craft on open models, fire at closed ones. It also produces gibberish suffixes, which makes the attack detectable by a perplexity filter (a defence we revisit in 9.5). Neither limitation lasted. Within months, AutoDAN (Zhu et al., 2023) added a fluency term to the GCG objective and produced readable, low-perplexity suffixes that pass the perplexity filter, and PAIR (Chao et al., 2023) dropped the gradient requirement entirely: an attacker LLM refines a semantic jailbreak against the target through black-box queries alone, typically succeeding in under twenty. So an attacker today needs neither open weights nor a transfer step. What this course takes from GCG: a jailbreak is a search with an objective, so "we patched the prompts we found" is never a defence against the next run of the optimiser.

The African-lens angle: tokenisers are not neutral

GCG is optimised over an English-trained model's vocabulary, and its cost scales with how many token-positions the search must cover. Low-resource languages are tokenised far less efficiently: isiZulu's agglutinative morphology fragments into many sub-word pieces, so the same sentence costs several times more tokens than its English translation. That changes the attack surface in both directions: more tokens mean a larger search space for an attacker, but also a longer, stranger input that a safety filter has even less training data to judge. The interaction of tokenisation, morphology, and robustness in African languages is wide open, a natural project seed once you have run the 9.4 lab.

Questions to bring to class

  • Write the GCG objective for a target of your choice. Why optimise the negative log-likelihood of an affirmative prefix rather than directly maximising "harmfulness"? What does that tell you about what is easy vs hard to specify?
  • In step 1 the gradient ranks \(|V|\) candidate tokens per position; in step 2 only a sampled batch is evaluated exactly. Why not trust the gradient's top choice? Why not exactly evaluate all \(|V|\) candidates?
  • Given the distillation caveat, design a fairer experiment to compare two models' robustness to transferred GCG suffixes. What would you change about the surrogate?
  • A perplexity filter rejects high-perplexity (gibberish) inputs. Does that defeat GCG, or just move the optimiser's objective? Write the new objective the attacker would minimise, then compare it with AutoDAN's (in the readings).

Readings

Core

• Zou, Wang, Carlini, Nasr, Kolter & Fredrikson (2023), "Universal and Transferable Adversarial Attacks on Aligned Language Models" (arXiv:2307.15043). §2 (the objective and Algorithm 1) and Tables 1–2 (the results above); the llm-attacks repo if you want to run it.

Supplementary

• Shin et al. (2020), "AutoPrompt" (arXiv:2010.15980). The gradient-guided prompt search GCG builds on.

• Zhu et al. (2023), "AutoDAN: Interpretable Gradient-Based Adversarial Attacks on Large Language Models" (arXiv:2310.15140). GCG plus a fluency objective: readable, low-perplexity suffixes that pass perplexity filters.

• Chao et al. (2023), "Jailbreaking Black Box Large Language Models in Twenty Queries" (PAIR) (arXiv:2310.08419). Semantic jailbreaks with black-box access only, typically in under twenty queries.
