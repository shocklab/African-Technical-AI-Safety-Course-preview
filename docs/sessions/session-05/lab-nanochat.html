Week 3 • Session 5.4

Lab: from base model to assistant (nanochat)

Read the real training pipeline in nanochat, then see (and make) the base → instruct difference yourself

What we'll cover

This lab turns the pipeline of 5.1–5.3 from words into code. You'll read the stages of a complete, minimal LLM pipeline (Karpathy's nanochat), then do the experiment that makes the session click: compare a base model (completes text) with an instruction-tuned model (follows instructions) on identical prompts, and, optionally, run a tiny supervised fine-tune yourself and watch behaviour shift. Everything graded runs on free Google Colab.

Setup Colab — no GPU needed for the core tasks

Open a new Colab notebook and install the Hugging Face libraries:

!pip install "transformers>=4.44" datasets torch

A free Colab account is the only requirement. Save the notebook; you'll submit it.

① Read the pipeline: nanochat

The whole stack in one readable repo

Andrej Karpathy's nanochat (github.com/karpathy/nanochat, MIT-licensed) is a minimal, end-to-end ChatGPT-style pipeline (tokenisation → pretraining → supervised fine-tuning → evaluation → inference → a web chat UI) designed to be read in full. You will not run the whole thing (it targets a multi-GPU node), but you should read it:

  • Find the pretraining stage: locate the next-token cross-entropy loss (5.1) in the training loop.
  • Find the SFT stage: see how it switches to (prompt, response) data and masks the loss to the response tokens (5.2).
  • Note what is not there in a from-scratch run: no preference/RLHF stage by default; that is Session 6.

Write 3–4 sentences mapping the repo's stages onto the five-stage table from 5.3. (The older, even smaller nanoGPT repo is a fine alternative for just the pretraining loop.)

② The experiment: base vs instruct

Same prompts, two models

Load a small base model and the instruction-tuned model built from it, and probe them identically. The pair below share architecture, size, tokeniser and pretraining data; the only difference is post-training, so whatever behavioural gap you find is the post-training stack of 5.3, isolated:

from transformers import pipeline
base = pipeline("text-generation", model="Qwen/Qwen2.5-0.5B")  # base
chat = pipeline("text-generation", model="Qwen/Qwen2.5-0.5B-Instruct")  # instruct

prompt = "List three causes of the First World War."
print(base(prompt, max_new_tokens=80))

messages = [{"role": "user", "content": prompt}]
print(chat(messages, max_new_tokens=80))
  • Note the two call shapes. The base model gets a raw string; the instruct model gets a messages list, which makes the pipeline apply its chat template (the special tokens marking user and assistant turns it was trained on, 5.2). Pass the instruct model a bare string and you are probing it outside its trained format; try it once and compare.
  • Give both the same three instructions (a question, a "summarise this", a "translate this"). Record what each does.
  • You should see the pattern from 5.1: the base model tends to continue the text (more questions, tangents) while the instruct model answers. Describe the difference in your own words; this is the SFT effect, made visible.
  • African-safety probe: repeat one instruction in isiZulu (or another language you read). Does the gap between base and instruct widen? Note what you see: it previews Session 9.
  • Optional era contrast: run the same prompts through gpt2 (2019, base). The gap is far larger, but it confounds era with tuning: gpt2 differs from the Qwen pair in data, size and architecture as well as post-training. It shows how far base models themselves have come, while the matched pair above isolates what post-training adds.

③ Optional: run a tiny supervised fine-tune [GPU — optional]

Fine-tuning on your own demonstrations

On a free Colab GPU runtime, fine-tune the small base model on a handful (10–50) of hand-written (instruction, response) pairs in one style (e.g. always answer in one short sentence) using the next-token loss masked to the response (5.2). Then re-run the prompts from ②. Even a tiny SFT set visibly nudges the model toward your demonstrated format. This is the mechanism behind "turning a base model into an assistant", in miniature; keep the dataset small and the effect will still show.

Session 5 summary & what's next

An LLM is built in stages: web-scale self-supervised pretraining yields a capable but unaligned base model (5.1); supervised fine-tuning on demonstrations makes it follow instructions but can only imitate, never select the better answer (5.2); the full post-training stack shapes the assistant as a thin proxy-driven layer over a vast base, with a distinct misalignment opening at every stage (5.3); and you've now seen the base→instruct shift in code (5.4).

Next (Session 6): the stage that breaks through SFT's ceiling, RLHF. We learn a reward model from human preferences, optimise the policy against it with a KL leash, and confront the price: optimising a learned proxy invites the Goodhart failures of Session 3.1: reward hacking and sycophancy.
